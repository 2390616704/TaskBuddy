# TaskBuddy 智能体协作说明

## 项目目标

按照 `全栈工程师入职考题_对话智能体页面基础功能设计.md` 实现一个可在本地运行的单智能体对话应用。工程交付必须完整、可解释、可复现，不以堆叠框架和功能数量为目标。

## 已确认的技术方向

- 前端：Next.js、React 和严格模式 TypeScript。以 Spring AI Alibaba Studio 的聊天界面为设计参考，但不得整块复制其代码。
- 后端：Python 和 FastAPI。
- 持久化：SQLite，并通过仓储接口隔离具体数据库实现。
- 传输：使用有明确事件类型的 SSE。浏览器取消请求时，取消信号必须传播到服务端生成任务。
- 模型执行：默认使用确定性 Mock Provider，确保没有 API Key 也能完整评审。
- 模型集成采用方案 A：业务代码只依赖项目自有的轻量 `ModelProvider` 协议；OpenAI Agents Python SDK 只能作为可选适配器，不得成为业务架构中心。

## 设计门槛

- 在设计被展示并确认前，不得开始实现产品代码。
- 第一版只实现一个“工作事项助手”，不得擅自加入多智能体编排、工具调用、认证、知识检索、附件、语音或云部署。
- 参考仓库只能作为设计依据。复制任何实质性代码前，必须核对许可证、记录来源，并解释针对本项目所做的改造。

## 必须遵守的模块边界

- UI 组件、客户端状态、HTTP/SSE 路由、应用服务、模型 Provider、Prompt 构建、输出校验和持久化必须位于职责清楚的独立模块中，并通过显式接口协作。
- 服务端是会话和消息持久化状态的唯一事实来源。客户端乐观状态必须使用服务端返回的标识和状态完成对账。
- 系统规则、历史消息、用户输入和外部上下文必须作为不同可信等级的输入进行 Prompt 构建和日志处理。
- 浏览器代码、日志、测试数据、截图和 Git 历史不得出现 Provider 密钥、完整系统提示词、私人会话或敏感环境变量。
- 取消操作必须幂等，必须停止下游生成，禁止继续保存迟到的增量，并最终持久化为 `cancelled` 状态。
- 重试必须产生可追踪的新生成尝试，不得重复保存用户消息。
- 所有 API 错误必须采用稳定结构，至少包含 `code`、`message` 和 `requestId`。

## Model Provider 约束

- 应用服务只能依赖项目自有的 `ModelProvider` 协议和内部事件类型，不得直接依赖 OpenAI Agents SDK 的类型。
- `MockModelProvider` 是默认实现，必须支持成功流式输出、主动取消、Provider 异常和结构化输出失败等确定性场景。
- `OpenAIAgentsProvider` 是可选实现，负责把 Agents SDK 的输入、流式事件、异常和最终输出转换为项目内部模型。
- Prompt Builder、历史截断策略和输出校验属于项目自身职责，不得隐藏在 Provider 或路由函数中。

## 必须提供的用户行为

- 支持创建会话、选择会话、加载历史、流式回答、取消、失败和重试。
- 明确区分发送中、流式生成中、已完成、已取消、失败和可重试状态。
- 支持 Enter 发送、Shift+Enter 换行、键盘操作、可读对比度，以及 375px 和 1440px 宽度下的可用布局。
- UI 必须渲染校验后的助手内容，不得直接展示 Provider 原始 JSON。

## 验证要求

- 自动化测试必须覆盖：创建会话、成功流式完成、取消、Provider 异常和 Prompt 注入拒绝。
- 声称完成前，必须运行最终 README 记录的格式化、Lint、严格类型检查、测试、构建和端到端冒烟命令。
- 必须验证：重启后持久化、浏览器断连、并发发送、结构化输出校验失败、重试幂等性和密钥/日志扫描。
- 未实际保留命令输出作为证据时，不得声称命令已经通过。

## 必须交付的文档

- `README.md`：架构、全新环境安装、启动、测试、构建，以及 Mock/真实 Provider 切换方法。
- `docs/prompt-design.md`：Prompt 分层、信任边界、上下文策略、输出 Schema 和三个必测行为场景。
- `docs/ai-collaboration.md`：真实记录 Codex 与 Claude Code 的关键 Prompt、采纳与拒绝的建议、人工判断和验证动作。
- `docs/redundancy-audit.md`：每个规定审计项的检查方法、发现、处理结果和证据位置。
