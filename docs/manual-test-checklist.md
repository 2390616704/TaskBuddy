## 长会话视口与滚动布局

- 状态：用户报告通过
- 前置条件：本地前后端已启动，已有一个包含足够多消息、内容高度超过视口的会话。

### 桌面端 1440px

1. 将浏览器视口宽度设置为 1440px，打开长会话。
2. 在消息区域内向上、向下滚动。
3. 将消息滚动到中间位置，观察页头、输入框和侧栏。
4. 创建足够多的会话，使侧栏列表高度超过视口，再滚动侧栏列表。

预期结果：

- 浏览器 `body` 不产生用于滚动整页的纵向滚动条；
- 只有消息区域滚动，Agent 页头和 Composer 始终留在视野中；
- 左侧栏高度固定为视口高度，不随消息内容增长；
- 会话项过多时仅侧栏列表滚动，品牌区和“新建会话”按钮保持可见；
- Composer 不覆盖最后一条消息，最后一条消息可以完整滚动到可见位置。

实际结果：用户于 2026-07-15 报告测试完成，未报告与预期不符的现象。

## Provider 按钮与 Markdown 流

- 状态：用户报告通过

1. 刷新页面并点击右上角模型按钮。
2. 选择 Mock，确认系统创建并切换到新的 Mock 会话。
3. 再次点击按钮，选择已配置的真实模型，确认创建新的真实模型会话。
4. 分别切回两个会话，观察右上角与输入区的模型名称。
5. 在 Mock 会话输入“请帮忙梳理一下上线可能存在的问题”。
6. 输入包含 Markdown 表格、链接、代码块的消息，观察显示。

预期结果：

- 右上角是可点击按钮，点击后展示服务端 Provider 列表；
- 未配置 Provider 显示“未配置”且不可选择；
- 选择 Provider 会创建新会话，不会修改已有会话；
- 每个会话刷新后仍显示自己的 Provider；
- 模型增量是自由 Markdown，不出现 JSON、`mode`、`sections` 等包装字段；
- 风险类改写仍出现“结论、风险项、待确认项、下一步”关键板块；
- 普通回答不被强制套用四个板块；
- 原始 HTML 不执行，外部链接安全地在新窗口打开。

实际结果：用户于 2026-07-15 报告测试完成，未报告与预期不符的现象。
