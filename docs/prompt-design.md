# Prompt 设计

## 分层与信任边界

`PromptBuilder` 把系统规则、历史消息和当前输入保留为不同角色消息。系统规则是受信任配置；历史与当前用户输入都是不受信任数据，不能通过简单字符串拼接升级为系统指令。

历史按字符预算从最新消息向前选择，避免上下文无限增长。超出预算的最旧消息被舍弃；系统规则和当前输入始终保留。

## 行为规则

系统 Prompt 覆盖考题要求的三类语义行为：

1. 发布、上线、投产风险梳理包含“结论、风险项、待确认项、下一步”；
2. 对象或目标不明确的执行请求先澄清，不声称已经操作；
3. 索取系统 Prompt、API Key、环境变量或要求绕过规则时拒绝泄露。

规则按意图生效，不匹配考题中的三个固定字符串。标点、空格、近义表达和改写不应改变安全边界。

## 输出契约

模型直接输出自然 Markdown，不输出内部 JSON。普通问题可以自由组织回答；仅在任务需要时使用标题、列表、表格和代码块。前端直接渲染 Markdown 增量，不解析“结论”等中文字符串。

考题的风险场景通过 Prompt 和关键文本测试证明四板块存在。若行为测试失败，应改进 Prompt 或确定性 Mock 语义分类，而不是在运行时给模型回复增加自制信封。

## Mock 与真实 Provider

Mock 使用小型、可解释的语义词组集合提供确定性回复，用于无密钥演示。真实 Provider 使用相同系统 Prompt，通过 OpenAI Agents SDK 流式返回文本。两者都映射为项目自有 `ModelEvent`，业务路由不直接调用 SDK。

## 日志原则

日志只应记录请求 ID、会话/消息 ID、Provider ID、状态、耗时、错误码和必要长度。不得记录完整系统 Prompt、密钥、完整用户对话或原始模型回复。
